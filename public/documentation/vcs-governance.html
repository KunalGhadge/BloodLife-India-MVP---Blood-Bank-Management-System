22. Managing Code with Git

We use Git to track every single change made to the codebase. This allows multiple people to work on the project at once without overwriting each other's work.

Concept: Branching. Think of the main code as a tree trunk. When you want to add a feature, you create a "branch," work on it, and eventually merge it back into the trunk.

Our Branching Strategy

  • Main: The stable version that is currently live. We never work directly on this branch.
  • Feature Branches: Used for adding new things (e.g., feature/inventory-search).
  • Bugfix Branches: Used for fixing specific issues (e.g., fix/donor-validation).

Commit Guidelines

To keep the history clean, we use "Conventional Commits." Your message should start with a prefix:

  • feat: for new features.
  • fix: for bug fixes.
  • docs: for documentation changes.
  • style: for UI/CSS changes.

Code Reviews

Before a branch is merged into main, it must be reviewed by at least one other developer. This ensures that the code is readable, efficient, and free of obvious bugs.

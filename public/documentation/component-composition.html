18. How the App is Built (Component by Component)

We believe in "Atomic Design." This means we build small, simple pieces (like buttons and inputs) and combine them into larger molecules (like a donor registration form) until we have a full page.

Principle: Single Responsibility. Each component should do exactly one thing and do it well.

Our Component Categories

  • Layout Components: These define the structural shell, like the Navbar and the Sidebar you see on this page.
  • Management Modules: Large, complex components that handle specific tasks like DonorManagement or InventoryControl.
  • UI Primitives: Tiny, reusable bits like Badge, Button, and Card. These ensure the app looks consistent everywhere.

Data Flow (Props)

Data flows "downwards" from parent components to child components using Props. This makes the app predictable—if you want to know where a piece of info came from, you just follow the trail up the component tree.

Reusability

Because our components are independent, we can reuse the DonorCard in the "All Donors" list AND in the "Matching Results" list without writing any new code!
